    Design the architecture, name the components, document the details

    23 Dec 2022 » go

    About this post

    The Go programming language is easy to learn. Go has only about 25 reserved words as compared to Java 67 (not including frameworks) or C# 79. The Go toolchain to compile Go codes, manage dependencies, generate documentation and perform the analysis is also very easy to use.

    After my initial and short exposure to Go, I was very comfortable using the language. I felt ready to work on a production-grade project using Go. However, I found architecting Go projects that promoted clean, clear, concise and maintainable code a challenge. When I embarked on my first project, there was little consensus on good architecture for Go projects. So I applied software engineering paradigms that I had learnt from my previous working experience, which was a mix of object-oriented and, sort of, functional programming. I had assumed that the paradigms were transferrable.

    Before using Go, I worked on projects based on Java, Objective-C, Swift, and Javascript. Applying the paradigms from these languages directly, I found, belatedly, as I evolved my Go projects, they became too complex and hard to maintain. I had layered stuff that I later discovered was unnecessary.

    To avoid that pitfall in future projects and use the Go language as intended, I decided to pivot and have a clean slate. So I delved into the philosophy behind the language’s creation and I turned to a presentation by Rob Pike, one of the creators of the language, where he presented the Go Proverbs.

    The Go Proverbs gave me the necessary philosophical foundation to break from my past and exploit the goodness of Go which I will share my thoughts in a series of blogs. In this post, I will address the aspect of the proverb titled “design the architecture, name the components, document the details”.

    Go packages

    A Go package is the basic building block of a project. It is a folder containing one or more source files. A Go source file has the extension *.go. The Go source files are organised as either part of a main package or a named package.

    A main package contains a source file or files with the signature package main and one main function. The folder containing such files can have names different from its package signature. There is no requirement for these folders to be specifically named main.

    A named package contains a source file or files with a signature that matches its containing folder. For example, source files with a signature package something must be found under this folder named something, like so:

      /something
    source.go <-- contain "package something" at the top of the file

    With some exceptions that I will not discuss here, Go is not opinionated about naming packages or naming source files in each package, thus I found myself confronted with these questions:

    • How should I group my source files into a Go package?
    • How should I name my Go packages and their source files?
    • If we have more than one package, how should I aggregate them?
    • How do I export or avoid exporting Go packages for used by other projects?
    • How do I use Go packages and their contents to model the solution my project is intended to implement concisely whilst minimising complexity?

    Deciding what to include in and naming a Go package

    Let’s consider object-oriented programming using Java as an example. This is to give you a sense of how I applied what I discovered applying object-oriented paradigm (i.e. Java) to my early Go projects.

    A basic unit of Java code is an object. An object has a data field and method attached to it. I specify the structure (data fields and methods) of an object via a class. Java, and other object-oriented languages, have the concept of static data fields and member methods that are part of the class scope. A Java class (public) must match the name of the class it contains. I could group related Java classes under a Java package (some object-oriented languages refer to this as a namespace).

    On the whole, I found Java opinionated when naming and packaging things. When you name things, the project structure (director and files) are arranged and named accordingly. I also learned there is a broad consensus that Java is suited to Domain-Driven Design methodology. For example, you might find a close correlation between a Java class and an Entity in the context of Domain-Driven Design. And when I was working on Java projects, I applied Domain-Driven Design, often unconsciously, when architecting my projects.

    In the case of Go, the equivalent of a class is known as a struct where I could associate it with functions to create kind of member methods1. However, unlike Java classes, a Go struct does not support inheritance or polymorphism. After suffering many Go projects pains, I belatedly discovered it was possible to declare a function type and associate that type to a method. Kind of like giving a function-type member method. I would use the feature to override the behaviour of a function to mimic Java member methods override. I attribute not knowing the Go feature to my dogmatic application of the object-oriented viewpoint. It is beyond the scope of this blog to explain how I use the function type method. I will discuss this feature in future blogs. Suffice it to say using that feature made my code more concise.

    Also unlike a Java class, a struct name need not match the file name that contains it.

    A Go package name and its content have a coupled relationship. So if I wanted to consume the function Now() in the standard library package time, I will have to call it this way time.Now(). I learned that I could use this coupling property to mimic the concept of static variables and methods found in Java. But I also found using it that way had lots of unintended consequences when used in the context of concurrency.

    When it comes to naming things in a Go package, I learned that it is not as opinionated as Java. So the early days of my Go coding journey, I often had to ask myself how to name things that do not make my code hard to read. To address this question, I follow the official Go package naming convention described in the Go blog, and I will summarise here2:

    • package names are lowercase with no underscore or mixed caps (but the Go compiler will accept these names);
    • package names are short and clear often simple nouns;
    • avoid ambiguous names, such as util or common, be specific;
    • avoid repeating the content and package name, for example, the server – e.g. function name – provided by the package http should be called Server not HttpServer so when the function is called, it will be http.Server instead of http.HttpServer which would suggest that under the http package, there might be other http server.

    As part of the exercise in naming things, I needed to identify things I wanted to build, and this meant I needed to decide what I wanted my Go package and its content to represent. I learned there was no consensus in the Go community on ways to model things.

    For small projects, my approach was to group related code elements (structs, functions, etc.) under Go packages on an ad-hoc basis. I would start with a single Go package and then refactor as the project evolved. However, for large projects such as building microservices and involving a team of developers, this technique was hard to practice.

    The Domain-Driven Design methodology is a popular approach for projects building microservices. However, there are some reservations amongst the Go community about the applicability of this methodology. Nevertheless, I use some aspects of Domain-Driven Design methodology to guide me when creating Go packages and their content. It gave my project a consistent structure, but I had to apply the technique cautiously. As Go is not a fully object-oriented programming language adopting Domain-Driven Design could compromise the feature of Go that makes it easy to create concurrent solutions. As to why I found it necessary to use Domain-Driven Design methodology carefully, it is a topic that requires a separate blog. The reasoning requires a nuanced explanation that is not appropriate for this blog.

    Organising your project structure

    Often I need to organise my Go packages to exist alongside non-Go-related artefacts, such as deployment scripts, build scripts, etc. I may also need to export my Go packages or not export my Go packages. These are decisions related to project layout. There are two types; library-oriented and service-oriented projects.

    A library-oriented project is intended purely for sharing Go artefacts with other projects. Service-oriented projects are intended to build one or more executable binaries. There is no standard way of laying out a service-oriented or library project. However, I found a template controversially known as the Standard Go Project Layout3. The template offers clues to how to structure your project:

    • It offers features that you can adapt to a library-oriented or service-oriented project.
    • It can accommodate non-Go artefacts (build script or other languages) alongside Go packages whilst maintaining a clear separation of purpose.

    Library-oriented

    A Go library (also known as a Go module) is a collection of Go source codes or Go packages I use to share with other projects. There are three main types:

    • a single package library;
    • a multi-package library;
    • a generated library.

    In the case of a single package library, I treat my Go package as the equivalent of the root of a source code repository. The source files are laid out flat in the code repository with its name for the Go package name. You can see an example in Corbra library.

    A multi-package library is a collection of Go packages. I could lay out the Go packages in a code repository :

    <repository>/
  fibgen/  <-- Go package containing functions to generate fibonacci sequence
  cryptoutil/ <-- Go package containing functions to generate cryptographic keys
  README.md
  go.mod

    However, I prefer a layout using the pkg arrangement described in the Standard Go Project Layout. So I would lay out my Go packages to look like this:

    <repository name>/
  docs/
  examples/
  pkg/
    fibgen/
    cryptoutil/
  README.md
  go.mod

    My preferred layout has the advantage of differentiating Go packages under pkg folder from docs containing markdowns or examples containing runnable examples. I tend to use this layout where I generate the library codes from some third-party specifications such as protobuf. My project layout would look like this:

    <repository>/
  build/
  docs/
  pkg/
    pb/  <-- generate Go package from protos
  protos/
    model.proto <-- protobuf spec
  /scripts <-- script to kick off the build process
  README.md
  go.mod

    Service-oriented

    In the case of service-oriented layout for projects building one or more executable binaries, one approach to laying out a project is:

    <repository>/
   utility.go
   main.go <-- contain only main function
   go.mod

    However, this is only for small projects involving only one main package.

    A more realistic project will involve not just Go codes but also non-Go artefacts like build or deployment scripts. Some projects may also involve multiple executables running in a platform or clusters of containers. For this type of project, I would layout the project like this:

    <repository>/
  build/
  deployment/
  cmd/ <-- folder containing multiple main packages
    service_a/
        internal/ <-- Go packages primarily for service_a only
          restapi/
      main.go
    service_b/
        internal/
          restapi/
      main.go
  docs/
  internal/ <-- share but not exportable Go packages for service_a and service_b
    fibgen/
    cryptoutil/
  scripts/
  README.md
  go.mod

    I could also combine Go with other languages. Please check out my example here. It demonstrates a service-oriented project that combines Go with ReactJS source codes.

    Document the details

    Go emphasises short package names. The consensus amongst the Go community is that it keeps code compact. Some have argued that naming Go packages with short names is not helpful as readers cannot glean its context. Fortunately, the Go toolchain makes it easy to document and add details to Go package names and their content.

    I can add detail about a Go package via godoc style documentation in Go codes. Here is an example of documentation about a Go package:

    // Package sort provides primitives for sorting slices and user-defined
// collections.
package sort

    The same applies to the content of a Go package. I could document shown in this example:

    // Fprint formats using the default formats for its operands and writes to w.
// Spaces are added between operands when neither is a string.
// It returns the number of bytes written and any write error encountered.
func Fprint(w io.Writer, a ...interface{}) (n int, err error) {

    The godoc format is not the only way to present details about a Go package and its contents. Readers might prefer information in other forms, as Rob Pike noted in another Go proverb, documentation is for users. With a few keystrokes of the godoc command, part of the Go toolchain, documentation about packages can be displayed in a web browser (see Figure 1) for a more human-readable and searchable form.

    The same information about a specific Go package can also be displayed from the godoc cli (see Figure 2).

    I found using godoc formatting a great way of enabling others to learn about my Go packages and their content and is sufficient for library-oriented projects. However, for service-oriented projects where I had to build artefacts, perform tests and deploy, I would include a README.md and a docs folder following the Standard Go Project Layout template. Sometimes, I find it useful also to add examples to help others who are maintaining my code appreciate the content. My recommended layout is this:

    <repository>/
  docs/
    deployment.md
    test.md
    build.md
  examples/
  README.md

    Conclusion

    To summarise, for any Go projects and putting Rob Pike’s proverb on architecture, naming things and documentation into action, I recommend these principles when you work with Go:

    • Use short, clear, and concise names for your Go packages and their content.
    • Do not replicate paradigms from other languages blindly.
    • Use the Standard Go Project Layout as the guide to lay out your project.
    • Domain-Driven Design methodology can help you structure your project but use it carefully and not compromise Go’s concurrency feature.
    • Use godoc and markdowns to document things.

    Notes

    1. Go technically does not have member methods like other object-oriented prorgamming language. It is a function with the associated type as a parameter.
    2. Out-of-box Go toolchain does not enforce the official Go package naming convention.
    3. Despite the name Standard Go Project Layout, it is not an official standard created by the Go core language developement team. You should not assumed that the Go toolchain is going to enforce this layout. The template probably drew inspiration from the hexagon architecture pattern presented by Alistair Cockburn.
